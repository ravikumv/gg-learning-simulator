What is the GoldenGate REST API?

Every microservice in the GoldenGate Microservices Architecture (MA) — Service Manager, Administration Service, Distribution Service, Receiver Service, and Performance Metrics Service — publishes its own REST API. Anything you can do through the web UI or AdminClient is, under the hood, a REST call to one of these services.

🔌 It's the control plane, not a bolt-on

AdminClient — the modern replacement for classic GGSCI — doesn't talk to Extract/Replicat directly. It issues REST calls against the Administration Service's /services/{version}/... endpoints. The REST API is the interface; AdminClient, the HTML5 web UI, and OCI GoldenGate's console are all just clients of it.

🧩 One base path, many services

Each deployment exposes its services on their own ports/paths, all following the same versioned pattern: /services/{version}/<resource>. JSON is the payload format throughout, and HTTPS with basic auth or client certificates is the standard security model.

Why automate through the REST API instead of AdminClient scripting?

AdminClient scripts are still shell-bound and host-specific. The REST API lets any language (Python, Java, Go, PowerShell) or any platform (CI/CD runner, Kubernetes operator, ServiceNow workflow, monitoring agent) drive GoldenGate the same way — asynchronously, remotely, and without needing OGG_HOME on the caller's machine. Long-running actions (like creating a deployment) return a background request you poll via /services/{version}/requests/{request} instead of blocking a session.

🟢 Discoverable

Call GET /services to list supported API versions, then GET /services/{version}/metadata-catalog for a full self-describing catalog of every resource the deployment exposes.

🔐 Secured per-role

Users are bound to roles (Administrator, Operator, Security, User) via /services/{version}/authorizations. The same endpoint enforces exactly the same permission model as the UI and AdminClient.

📊 Metrics-native

The Performance Metrics Service exposes dozens of endpoints under /mpoints/{item}/... — trail I/O, queue stats, replicat batching, network throughput — built for dashboards, not screen-scraping.

The Service Manager is the single entry point per host; it fronts one or more deployments, and each deployment's services expose their own REST surface underneath it.

Administration Service

The workhorse for automation: create/patch/delete Extracts and Replicats, issue start/stop commands, manage trails, supplemental-logging (trandata), credentials, encryption keys, and heartbeat tables. This is the service AdminClient talks to for almost every command you'd have typed into classic GGSCI.

Distribution & Receiver Services

Distribution Service owns the source-side path that ships trail data off-box (/sources/{distpath}); Receiver Service owns the target-side collector that accepts it (/targets/{path}). Together they replace the classic data-pump/collector pair with REST-managed paths.

Performance Metrics Service (PMSRVR)

Every Extract, Replicat, and Distribution/Receiver path reports into PMSRVR. It's a separate collection layer from the Administration Service — pulling status doesn't touch the process control path, so dashboards polling metrics endpoints don't add load to the thing they're monitoring.

Where teams actually use the GoldenGate REST API

Anywhere GoldenGate needs to be driven by something other than a human clicking through a console.

CI/CD

Pipeline-driven deployment promotion

A release pipeline calls POST /deployments/{deployment} and POST /extracts/{extract} to stand up an identical Extract/Replicat topology in QA, then tears it down after the test run — no manual GGSCI/AdminClient session needed.

Monitoring & ITSM

Feeding Grafana, Splunk, or ServiceNow

A poller hits GET /extracts/{extract}/info/status and the PMSRVR lag/throughput endpoints on a schedule, pushing results into existing dashboards or auto-opening an incident when lag crosses a threshold.

Multi-cloud orchestration

OCI GoldenGate inside Azure/AWS regions

Provisioning automation for OCI GoldenGate's managed service (including deployments physically hosted inside an Azure region via Multicloud Resource Anchor) uses the same REST surface behind the console to create deployments and wire up connections.

Self-service portals

Internal "request a replication path" UI

A lightweight internal web app lets application teams request new source→target paths; the backend translates the form into REST calls (create connection, add trandata, create Extract/Replicat/path) instead of giving every requester AdminClient access.

Disaster recovery automation

Scripted failover runbooks

A DR runbook calls the REST API to pause Replicats, validate checkpoints via /replicats/{replicat}/info/checkpoints, and resume against a new target — repeatable and auditable instead of a manual checklist.

AI / chat-ops

Natural-language operational assistants

The 26ai Admin Server AI Management endpoints (/aiservice/models) let an assistant translate "why is REPFABRIC1 behind" into the right status/log/metrics calls and summarize the result for an operator.
