Core Concepts

OCI GoldenGate is a fully managed, native OCI service for real-time data replication (CDC), streaming, and batch data movement. These are the building blocks you configure before any data moves.

📁 Compartment

Logical boundary that organizes and isolates OCI resources — networks, deployments, connections. Only deployments and database registrations within the same compartment can reference each other; it is also a security boundary.

🔗 Connection

Stores the network connectivity details (host, port, credentials, wallet/TLS) for a source or target. Connections are created and edited only from the OCI Console — never from the deployment console's Credentials screen, since changes sync down automatically.

📦 Deployment

A managed container that runs your OCI GoldenGate microservices (Extract/Replicat processes, trail files, deployment console). Sized in OCPUs, with License Included or Bring-Your-Own-License (BYOL), and optional auto scale.

🔐 Deployment Console

Web UI launched per-deployment. Tabs include Administration Service (manage Extract/Replicat), Distribution Service (send trail data across the network), Receiver Service (accept incoming trail data), and Performance Metrics Service. Note: the on-prem Service Manager is not exposed in OCI GoldenGate.

⚖ AdminClient (not GGSCI)

Administration is performed through the browser-based deployment console or the REST-based AdminClient — the classic GGSCI command-line interface is not how OCI GoldenGate deployments are administered.

📊 OCPU & Auto Scale

Deployments are billed and sized by OCPU. You can scale manually or enable auto scale at creation/edit time so the deployment grows or shrinks with load.

Ravi's convention: this simulator uses precise OCI terminology (Deployment, Connection, Peer Deployment, AdminClient) rather than legacy on-prem terms, matching current Oracle documentation.

Deployment Types

All OCI GoldenGate deployment types run on a single, natively integrated platform and are centrally managed in OCI — but each is provisioned as its own deployment resource.

🔁 Data Replication

Real-time Change Data Capture (CDC) replication using Extract and Replicat. The workhorse deployment type — supports Oracle, MySQL, SQL Server, PostgreSQL, Db2 z/OS, Big Data targets, and GoldenGate Distribution/Receiver paths. Only Data Replication deployments support manual and automatic backups and peer deployments for DR.

🧩 Data Transforms

Batch integration (ELT/ETL) for building data pipelines into analytics, data science, and lakehouse targets (e.g. Autonomous Database, Exadata Cloud Service, Snowflake). Uses Data Loads and Data Flows, optionally driven by a Data Replication deployment for continuous loads.

📡 Stream Analytics

Consumes streamed change data (via a Kafka connection and a GoldenGate connection) into pipelines with business-logic filters, then publishes results. Concepts: Connection, Stream, Pipeline, Business logic, Publishing.

✅ Data Verification Newer

Powered by GoldenGate Veridata. Compares and repairs data consistency between source and target so downstream consumers can trust replicated data — complements Data Replication, Data Transforms, and Stream Analytics rather than replacing them.

Topology rule Ravi enforces across the simulator suite: if source and target use the same technology family (e.g. two Autonomous Databases), one Data Replication deployment is usually enough. If source and target are different technology families (e.g. MySQL → Big Data), you typically need two deployments connected by a Distribution Path — and a Replicat delivers to exactly one target.

HA & DR — Cross-Region Switchover & Failover

A peer deployment is a standby that carries the same metadata (trail & parameter files, Block Volume, File Storage replicas) as its primary. A primary can have only one local-or-cross-region peer. Only Data Replication deployments support peer deployments today.

Switchover (Planned)

Manual, graceful role exchange — e.g. for maintenance or regional migration. Both deployments show Updating while roles swap. Duration depends on primary storage volume. Auto-start processes restart automatically on the new primary. No data loss expected since the exchange is orderly.

Failover (Unplanned)

Used when the primary region is unreachable (disaster). You promote the peer to primary without cooperation from the failed side. Because cross-region replication is asynchronous, a small, bounded RPO gap is possible — the new primary starts from the last replicated point.

Constraints Enforced Here

Exactly one peer per primary (local or cross-region, not both). Autonomous AI Database Serverless/Dedicated does not currently support database-level switchover in this flow. Switchover/failover is a manual action from the Disaster Recovery tab of either deployment's Details page.

Cross-region database switchover can surface log-mining errors (e.g. LogMiner unable to find a log file SCN) that are specific to certain database types — this simulator surfaces that as a warning during switchover, matching documented behavior.
