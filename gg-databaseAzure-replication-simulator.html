Where everything actually runs

Oracle Database@Azure hosts the databases; the GoldenGate deployment itself runs as an OCI-managed service, bridged into your Azure subscription through Resource Anchors and Network Anchors — it is not a VM you provision and patch, and it is not literally inside either database's compute.

So — Azure or OCI? Both, by design.

It is OCI GoldenGate software, but Oracle provisions the compute for it inside the Azure region you pick — the "Multicloud Partner Region / Partner Availability Zone" fields you choose during creation are literally naming a real Azure region and zone (e.g. East US, Zone 1). It isn't a remote OCI region you tunnel to. You just create and manage it from the OCI Console, and it's billed through your Azure subscription (ORACLEDBATAZURE) using Azure credits.

One deployment, cross-region

A single deployment can serve as both the Extract host and the Replicat host for a source/target pair even when the two databases sit in different Azure regions (e.g. source in UK, target in US East) — as long as the VNets/VCNs for both regions are peered, so all replication traffic still stays on the Azure backbone.

Prerequisites checklist

What has to be true before Extract and Replicat can be started for real.

Source — 19c CDB

  • ARCHIVELOG + FORCE LOGGING enabled at the CDB.
  • ENABLE_GOLDENGATE_REPLICATION=TRUE set in CDB$ROOT.
  • Supplemental logging enabled on replicated tables.
  • Common GG admin user (C##GGADMIN) granted via DBMS_GOLDENGATE_AUTH.
  • Source registered as a Connection inside the GoldenGate deployment (wallet/credential-based, via a private endpoint).

Target — Oracle Database 26ai

  • ENABLE_GOLDENGATE_REPLICATION=TRUE at target CDB$ROOT.
  • Target GG admin/apply user granted privileges; checkpoint table pre-created.
  • Target PDB open READ WRITE with a dedicated service name.
  • Target object structures already present (via Data Pump/initial load) before CDC starts.
  • Target registered as a Connection inside the GoldenGate deployment.

GoldenGate deployment

  • Resource Anchor and Network Anchor created in Azure, linking to an OCI compartment.
  • Deployment provisioned (Development/Test: 1 OCPU; Production: 4 OCPUs with autoscaling recommended).
  • Vault/Secrets policies configured for credential storage.
  • Replicat type decided (tab 04) — Coordinated needs manual range design; Classic/Parallel/Integrated do not.

Networking

  • VNets hosting both Database@Azure instances peered/reachable from the Network Anchor's delegated subnet.
  • Traffic path confirmed to stay entirely within the Azure backbone (no public internet hop).
  • Monitoring wired up across both the Azure and OCI consoles.

Core concepts

Three things happen, in order, for every change: capture, transport, apply.

Capture

Integrated Extract mines redo through the CDB's Logmining Server and writes Logical Change Records (LCRs) to a local trail file — container-tagged (CON_ID) so multitenant source changes stay attributable to the right PDB.

Transport

The Distribution Server on the deployment reads that trail and ships it — over the private Azure-backbone path enabled by the Network Anchor — to a Receiver Server, which writes an equivalent trail for Replicat to consume.

Apply

Replicat reads the trail and commits rows into Oracle Database 26ai. This is where Classic, Coordinated, Parallel Non-Integrated, and Integrated Replicat genuinely differ — same trail in, completely different apply mechanics (tab 04).

Network / Anchor topology

    
The mechanics behind "reachable from Azure, running in OCI."

    

      

        
Resource Anchor

        
    
          
  • A logical bridge between an Azure Resource Group and an OCI Compartment.
    • 
          
  • Keeps access, policy, and organizational alignment consistent across both clouds.
    • 
          
  • Required before an OCI GoldenGate deployment can be provisioned for Database@Azure.
    • 
        

      

      

        
Network Anchor

        
    
          
  • Maps an Azure VNet's delegated subnet into the OCI networking used by the GoldenGate deployment.
    • 
          
  • Keeps replication traffic on the Azure backbone rather than crossing the public internet.
    • 
          
  • The deployment is placed in a Multicloud Partner Region / Partner Availability Zone tied to this anchor.
    • 
        

      

      

        
Where you operate from

        
    
          
  • Deployment provisioning is currently initiated from the OCI Console only (Oracle Database → GoldenGate → Deployments), even though it's billed/organized under your Azure subscription via the anchors.
    • 
          
  • Example from Oracle's own walkthrough: a deployment named GGSAZ_ORCL provisioned with Multicloud Partner Region East US, Partner Availability Zone Zone 1 — meaning it runs in the Azure East US (Ashburn) region.
    • 
          
  • Day-to-day Extract/Replicat configuration happens through the per-deployment Deployment Console (Launch Console button), not the OCI Console itself.
    • 
        

      

      

        
Cross-region considerations

        
    
          
  • If source and target Database@Azure instances are in different Azure regions (e.g. source in UK, target in East US), the two VNets/VCNs must be peered first — one GoldenGate deployment can then still serve as both Extract and Replicat host.
    • 
          
  • Same-region deployments give the lowest latency; cross-region adds a small, generally predictable latency premium, still fully inside the Azure backbone.
    • 
        

      

      

        
Connection & config workflow (as actually done)

        
    
          
  • Create a Connection per database (wallet + password secret + GGADMIN credentials), then Assign both connections to the deployment and Test Connection on each.
    • 
          
  • Unlock the GGADMIN user on both databases and run ALTER PLUGGABLE DATABASE ADD SUPPLEMENTAL LOG DATA; on the source.
    • 
          
  • In the Deployment Console: add TRANDATA on the source connection, add a Checkpoint table on the target connection, then create the Extract and Replicat processes.
