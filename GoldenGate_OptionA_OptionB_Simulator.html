Downstream Extract for Oracle GoldenGate Deployment

Option A vs Option B topologies · Integrated Extract internals · Real-Time Capture Y/N behavior

Two ways to ship data downstream

Both options move committed change data from a source Oracle database to a downstream Oracle database using GoldenGate. The difference is where Extract captures from. Pick a card to jump into its live demo.

Option A — Direct Capture

Source Oracle DB ▸ Integrated Extract ▸ Trail File ▸ Network ▸ Replicat ▸ Data Lake House. Extract mines the redo stream directly on the source, in-database, via the integrated log mining server.

Lowest latency pathAdds capture overhead on source

Option B — Downstream Mining Database

Source Oracle DB ▸ redo/archive log shipping ▸ downstream mining database ▸ Integrated Extract ▸ Trail File ▸ Network ▸ Replicat ▸ target. The mining database holds no source data — it exists only so Extract can mine redo there instead of on production. A cascaded variant lets an Active Data Guard standby forward the redo to the mining database instead of the primary.

Zero capture load on sourceLatency depends on mining mode

Important: Extract can't capture from ADG directly

Per Oracle's documentation, Extract cannot connect directly to a Data Guard or Active Data Guard standby to capture data. When an ADG standby is in the picture, it only forwards redo to a separate downstream mining database — Extract always mines from that mining database, never from the standby itself.

What "Integrated Extract" means in both options

Integrated Capture registers Extract with the database's own log mining server (rather than reading raw redo files itself). The mining server assembles committed changes into Logical Change Records (LCRs) and streams them to Extract, which writes them into a proprietary trail file — the durable, replayable unit GoldenGate ships across the network. See the Integrated Extract & Trail File tab for the record-level walkthrough.

How the log mining server builds an LCR

  1. Redo generated by a commit is read from the online redo log buffer / log files.
  2. The in-database logmining server reconstructs each row change and assembles it into a Logical Change Record (LCR) — the operation type, table, SCN, and column values.
  3. LCRs stream to the Extract process over the database's internal streaming pool (no separate log-reading step, unlike classic capture).
  4. Extract writes each LCR as a record into the current trail file, advancing the Relative Byte Address (RBA) and periodically rolling to a new trail file at a configured size.

The distinction that matters

Real-Time Capture: Y

Extract mines changes continuously as they're generated — from the online redo log / log buffer via the integrated mining server — with no need to wait for a log switch or archival. Latency to the trail file is typically sub-second.

Real-Time Capture: N

Extract (or the mining server feeding it) is limited to reading redo only after it has been archived. Changes queue up until the next log switch, so latency tracks the log switch interval rather than the transaction itself.

On the downstream mining database (Option B), Oracle names these two modes explicitly: Real-Time Mining mode (processes online or standby redo logs immediately) and Archive-Log-Only mode (waits until logs are archived). This tab's Y/N toggle maps onto that same pair of modes.

Configuring a Downstream Extract

A condensed, walkthrough version of Oracle's downstream Extract setup — covering the direct mining-database path and the cascaded Active Data Guard variant. Source: Oracle GoldenGate 26 documentation, Downstream Extract for Oracle GoldenGate Deployment.

A downstream deployment offloads redo mining off the source database onto a separate mining database, which needs no source data — just an equal-or-higher database version, and the same endianness/processor family as the source. It's also the option to reach for when GoldenGate simply can't get a direct connection into the source database.

  1. Create the GoldenGate admin user on both databases

    A common user is required in the root container of each container database. On 21c and earlier, this is done with PL/SQL; on Oracle AI Database 26ai+ it's handled through the OGG_CAPTURE / OGG_APPLY roles instead.

    -- 21c and lower
CREATE USER c##ggadmin IDENTIFIED BY password CONTAINER = all;
ALTER USER c##ggadmin SET CONTAINER_DATA = all CONTAINER = current;
GRANT connect, resource TO c##ggadmin CONTAINER = all;

BEGIN
  DBMS_GOLDENGATE_AUTH.GRANT_ADMIN_PRIVILEGE(
    grantee => 'C##GGADMIN', privilege_type => 'CAPTURE',
    grant_select_privileges => TRUE, container => 'ALL');
END;
/

  2. Prep the source database

    Enable GoldenGate replication and archive logging, and turn on minimal supplemental logging (schema or table-level TRANDATA) since the source is still where the actual DML happens.

    ALTER SYSTEM SET ENABLE_GOLDENGATE_REPLICATION = TRUE;
ALTER DATABASE ARCHIVE LOG;
ALTER DATABASE FORCE LOGGING;   -- recommended, source only

  3. Prep the downstream mining database

    Same replication/archive-log settings, plus a streams pool on 21c and lower, and a different DBID / DBNAME / DB_UNIQUE_NAME / GLOBAL_NAME than the source. Password files must be shared/identical between the two.

    ALTER SYSTEM SET ENABLE_GOLDENGATE_REPLICATION = TRUE;
ALTER SYSTEM SET STREAMS_POOL_SIZE = 1G;   -- 21c and lower
ALTER DATABASE ARCHIVE LOG;

  4. Pick a mining mode: Real-Time vs Archive-Log-Only

    Real-Time Mining needs standby redo log groups on the mining database — at least one more group than the source has, sized to match the source's largest redo log. Only one real-time Extract per source database is allowed against a given mining database; any others must run Archive-Log-Only.

    -- add standby redo logs on the mining database (example: 3 groups)
ALTER DATABASE ADD STANDBY LOGFILE GROUP 3
  ('/oracle/dbs/slog3a.rdo','/oracle/dbs/slog3b.rdo') SIZE 500M;
ALTER DATABASE ADD STANDBY LOGFILE GROUP 4
  ('/oracle/dbs/slog4a.rdo','/oracle/dbs/slog4b.rdo') SIZE 500M;

-- redo transport + local archiving of standby logs on mining DB
ALTER SYSTEM SET LOG_ARCHIVE_CONFIG = 'DG_CONFIG=(CDBNORTH, cdbDSC)';
ALTER SYSTEM SET FAL_SERVER = 'CDBNORTH_root';
ALTER SYSTEM SET LOG_ARCHIVE_DEST_2 =
  'LOCATION=/home/arc_dest/srl_dbms1 VALID_FOR=(STANDBY_LOGFILE,PRIMARY_ROLE)';
ALTER SYSTEM SET LOG_ARCHIVE_DEST_STATE_2 = ENABLE;

    For Archive-Log-Only mode, skip the standby logs and instead set up ordinary redo transport from source to mining database:

    ALTER SYSTEM SET log_archive_config = 'DG_CONFIG=(CDBNORTH, cdbDSC)';
ALTER SYSTEM SET log_archive_dest_2 =
  'SERVICE=cdbDSC_root ASYNC NOREGISTER
   VALID_FOR=(ONLINE_LOGFILES,PRIMARY_ROLE) DB_UNIQUE_NAME=cdbDSC';

  5. Set credentials and Extract parameters

    The Extract parameter file needs a mining-database connection (MININGUSERALIAS), a source connection for fetches (FETCHUSERIDALIAS), and the real-time mining switch.

    ALTER CREDENTIALSTORE ADD USER c##ggadmin@CDBNORTH_root ALIAS cgg_CDBNORTH DOMAIN OracleGoldenGate PASSWORD password
ALTER CREDENTIALSTORE ADD USER c##ggadmin@cdbDSC_root  ALIAS cgg_cdbDSC   DOMAIN OracleGoldenGate PASSWORD password

-- extract parameter file
EXTRACT EXTDSC
USERIDALIAS cgg_CDBNORTH
TRANLOGOPTIONS MININGUSERALIAS cgg_cdbDSC
TRANLOGOPTIONS INTEGRATEDPARAMS (DOWNSTREAM_REAL_TIME_MINE Y)
FETCHUSERIDALIAS cgg_CDBNORTH DOMAIN OracleGoldenGate
EXTTRAIL Downstream/ea
SOURCECATALOG CDBNORTH_PDB1
TABLE HR.*;

  6. Register and start the Extract

    DBLOGIN USERIDALIAS cgg_CDBNORTH
MININGDBLOGIN USERIDALIAS cgg_cdbDSC

ADD EXTRACT extdsc, INTEGRATED TRANLOG, BEGIN NOW
REGISTER EXTRACT extdsc, DATABASE CONTAINER (CDBNORTH_PDB01)
ADD EXTTRAIL Downstream/ea, EXTRACT extdsc
START extdsc

  7. Optional: cascade through an Active Data Guard standby

    Use this when GoldenGate can't reach the source at all. The standby ships redo onward to the mining database, and — critically — Extract still never connects to the standby to capture data; it only uses ADG for read-only metadata fetches, started with the sourceless (NOUSERID) option.

    -- on the ADG standby: forward standby redo to the mining database
ALTER SYSTEM SET LOG_ARCHIVE_DEST_3 =
  'service=mining_db_service_name ASYNC NOREGISTER
   VALID_FOR(STANDBY_LOGFILES,STANDBY_ROLES) DB_UNIQUE_NAME=3rd_db_unique_name';
ALTER SYSTEM SET LOG_ARCHIVE_CONFIG = 'DG_CONFIG=(CDBNORTH, cdbADG_src, cdbDSC)';

-- extract parameter file, sourceless
EXTRACT EXTDSC
NOUSERID
TRANLOGOPTIONS MININGUSERALIAS cgg_cdbDSC_src DOMAIN OracleGoldenGate
TRANLOGOPTIONS INTEGRATEDPARAMS (DOWNSTREAM_REAL_TIME_MINE Y)
FETCHUSERIDALIAS cgg_cdbADG_src DOMAIN OracleGoldenGate
EXTTRAIL cascade/ea
TABLE HR.*;

DBLOGIN USERID ggadmin@cdbADG_src, PASSWORD ggadmin
MININGDBLOGIN USERID ggadmin@cgg_cdbDSC, PASSWORD ggadmin
ADD EXTRACT exte, INTEGRATED TRANLOG, BEGIN NOW
REGISTER EXTRACT exte DATABASE

Constraints worth remembering

• Source and downstream servers must share endianness and processor family (Linux x86 → Windows x86 works; HP-UX → AIX does not).

• Only one real-time Extract per source database can run against a given mining database — every other Extract against that source must be Archive-Log-Only.

• Extract can never connect directly to a Data Guard or Active Data Guard standby to capture data — the standby only forwards redo onward.
