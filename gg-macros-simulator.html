What is a GoldenGate Macro?

A GoldenGate macro is a named, reusable block of GoldenGate parameter-file syntax. Define it once with DEFINE MACRO #name, give it optional parameters, and CALL it inside TABLE or MAP statements. GoldenGate expands every CALL in-place at process startup — pure text substitution before execution begins.

Think of macros as managed copy-paste with parameters. One definition, called for every table. One change propagates everywhere instantly.

💡 Analogy

Without macros: adding 4 audit columns to 80 tables = 320 lines of COLMAP, repeated 80 times. With one #audit_cols macro: 80 CALL statements. If a column name changes, update one macro body.

Macro Anatomy — Every Part Explained
dirprm/gg_macros.mac — annotated
DEFINE MACRO  #audit_cols           -- ① keyword + name (must start with #)
PARAMS (                               -- ② optional parameter list
  &src_schema,                         --    & prefix marks a parameter
  &src_table,
  &tgt_schema,
  &tgt_table
)
BEGIN                                  -- ③ body start
  COLMAP (
    USEDEFAULTS,
    GG_OP_TYPE    = @GETENV ('LASTERR',       'DMLTYPE'),
    GG_COMMIT_TS = @GETENV ('GGHEADER',      'COMMITTIMESTAMP'),
    GG_EXTRACT_NM= @GETENV ('GGENVIRONMENT', 'GROUPNAME'),
    GG_TRAIL_SEQ = @GETENV ('GGHEADER',      'LOGRBA')
  )
END;                                   -- ④ body end — semicolon required

-- ⑤ CALL in Replicat MAP statement:
MAP HR.EMPLOYEES, TARGET HR.EMPLOYEES,
  CALL #audit_cols (HR, EMPLOYEES, HR, EMPLOYEES);
--                   ↑src_schema ↑src_table ↑tgt_schema ↑tgt_table
-- ⑥ Result: macro body expands in-place at startup
① DEFINE MACRO #name
Registers macro in GG's in-memory dictionary. The # prefix is mandatory. Name must be unique within scope.
② PARAMS (&name, ...)
Optional parameter list. Reference inside body with &param_name. CALL replaces each &param positionally.
③④ BEGIN ... END;
Body can contain COLMAP, FILTER, TABLE, MAP, TOKENS, SQLEXEC. END; semicolon is required.
How to Use Macros — Step by Step
1
Create a macro library file (.mac)
Create dirprm/gg_macros.mac. Write all DEFINE MACRO blocks here. Shared between Extract and Replicat — single source of truth.
2
INCLUDE the library at the top of your .prm file
Add INCLUDE ./dirprm/gg_macros.mac before any CALL statements. GG reads this and registers all macros before processing TABLE or MAP statements.
3
CALL the macro inside TABLE or MAP
Use CALL #macroname (arg1, arg2, ...) inside TABLE or MAP. Arguments are positional — must match PARAMS list in order and exact count.
4
Verify expansion with VIEW PARAMS
Run VIEW PARAMS <groupname> in GGSCI. Shows the fully expanded parameter file — every CALL replaced with its expanded body. Best macro debug tool.
5
Nest macros (call one macro from another)
A macro body can CALL another macro. Define inner macros first (top-to-bottom matters). No recursion — a macro cannot call itself.
Macro Rules — Do's
✓ DEFINE before CALLMacro must be defined or INCLUDEd before its first CALL — GG parses top-to-bottom.
✓ & prefix for paramsAlways use &param_name inside body. Without &, text is literal — not substituted.
✓ END; with semicolonOmitting the semicolon after END causes a parse error on startup.
✓ VIEW PARAMS to debugShows the fully expanded file in GGSCI after all macro substitution.
Macro Rules — Don'ts
✗ No recursionA macro cannot CALL itself. GoldenGate does not support recursive expansion.
✗ No runtime logicMacros expand at startup only. Use FILTER/SQLEXEC for conditional runtime logic.
✗ Argument count must matchCALL arg count must exactly match PARAMS count — mismatch = startup error.
✗ No $ or : prefixParameters use & not $ or :. Wrong prefix = no substitution occurs.

Step-by-Step Flow — Click to Expand
1
Source Oracle 19c — Supplemental Logging & LogMiner Registration
▼

GoldenGate Integrated Extract registers with Oracle 19c's internal LogMiner server (REGISTER EXTRACT ... DATABASE). Captures INSERT, UPDATE, DELETE, DDL from redo logs as LCRs. Supplemental logging must be enabled at DB and table level. The mining user requires LOGMINING and SELECT ANY TABLE privileges.

SQL + GGSCI — Oracle 19c setup
-- Enable supplemental logging
ALTER DATABASE ADD SUPPLEMENTAL LOG DATA;
ALTER TABLE HR.EMPLOYEES ADD SUPPLEMENTAL LOG DATA ALL COLUMNS;
-- GGSCI
DBLOGIN USERID ggadmin@ORCL19C, PASSWORD Welcome1
ADD EXTRACT      EXTE1, INTEGRATED TRANLOG, BEGIN NOW
REGISTER EXTRACT EXTE1, DATABASE
ADD EXTTRAIL     ./dirdat/ea, EXTRACT EXTE1, MEGABYTES 500
2
Macro Library — DEFINE MACRO in gg_macros.mac
▼

Shared macro library (dirprm/gg_macros.mac) holds all DEFINE MACRO blocks. Both Extract and Replicat load it with INCLUDE. Macros expand at startup via text substitution — parameters use & prefix. VIEW PARAMS shows the expanded result.

Key point: Macros are startup-time text substitution — NOT runtime. This makes them fast and predictable but means no conditional logic inside a macro body.
3
Trail Files — Canonical Format, Multi-Reader
▼

Trail files store before/after images, operation type, commit TS, and transaction context in GoldenGate binary canonical format. Both REPO1 (Oracle 26ai) and REPLK1 (Lakehouse) read the same trail concurrently. Always use FORMAT RELEASE 12.3 for full column metadata required for Parallel Replicat and Oracle 26ai targets.

4
Non-Integrated Parallel Replicat — Direct SQL Apply to 26ai
▼

ADD REPLICAT ... NONINTEGRATED PARALLEL creates N applier threads (PARALLELISM N) sharing a coordinator. Applies DML directly via OCI — no Oracle apply server needed. Macros in the .prm inject #vector_tag (AI pipeline), #scd2_stamp (history), #audit_cols (lineage) per MAP statement.

PARALLELISM N: Start with 4–8 for Oracle 26ai targets. Use BATCHSQL BATCHTRANSOPS 1000 to reduce round-trips. Tune dynamically via SEND REPLICAT <name> PARALLELISM N.
5
Oracle 26ai Target — AI-Ready Schema with Macro-Injected Columns
▼

Oracle 26ai adds native VECTOR columns, JSON Relational Duality Views, and AI Optimizer integration. The #vector_tag macro marks CDC rows with VECTOR_PENDING='Y' — a Scheduler job calls DBMS_VECTOR.EMBED() on changed rows, keeping 26ai vector indexes synchronized with Oracle 19c OLTP changes in real time.

Oracle 26ai + GoldenGate: VECTOR_PENDING='Y' + DBMS_VECTOR.EMBED() + LAST_CDC_TS watermark = real-time RAG search powered by GoldenGate CDC. All triggered by a two-line macro called once per table.
