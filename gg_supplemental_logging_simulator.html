📜 Standard Redo Log — WITHOUT Supplemental
⚠ Missing Data
Oracle Default Redo Entry
-- Standard redo only logs CHANGED columns UPDATE HR.EMPLOYEES SET SALARY = 95000 WHERE EMPLOYEE_ID = 101; -- Redo vector written by LGWR: OP: 11.4 (update) SCN: 0x0B.A43C21 col 5[SALARY]: 95000 ← after-image only Row identity: UNKNOWN ← problem! Before-image: NOT CAPTURED
❌
GoldenGate Extract Cannot Proceed
Without the primary key in the redo vector, Extract cannot identify which row changed. It must scan the full table or skip the operation — both are unacceptable for real-time replication.
🔑 Supplemental Redo Log — WITH Logging Enabled
✔ Complete
Enhanced Redo Entry
-- Supplemental log appends extra vectors UPDATE HR.EMPLOYEES SET SALARY = 95000 WHERE EMPLOYEE_ID = 101; -- Enhanced redo vector: OP: 11.4 SCN: 0x0B.A43C21 col 0[EMPLOYEE_ID]: 101 ← PK ✔ col 4[SALARY_BI]: 85000 ← before-image ✔ col 5[SALARY]: 95000 ← after-image ✔ col 1[EMAIL]: john@corp.com← UK col ✔
✅
Extract Builds Complete LCR
All pieces present: row identity (PK=101), before-image (85000→95000), key context. Trail file written. Replicat applies the change precisely at the target.
🏗️ Two-Level Supplemental Logging Architecture
🔴 Level 1 — Database Level (Minimal Supplemental Logging)
📋 Enables Redo Log Infrastructure
🔧 ALTER DATABASE ADD SUPPLEMENTAL LOG DATA
✔ Required Foundation for GGS
📌 Very low overhead
⬇ Enables table-level log group creation
🟡 Level 2 — Table / Schema Level (ADD TRANDATA)
📊 Per-table supplemental log groups
🔑 Primary / Unique key columns
📝 Before-image capture
🛡️ ALLCOLS for CDR
🏗️ SCHEMATRANDATA for bulk
⬇ Extract reads enhanced redo via LogMiner / Integrated mode
🟢 GoldenGate Extract — Trail File Production
🔍 Redo log mining (LogMiner API)
📦 LCR construction
🚀 Trail file write
↔️ Downstream Replicat

🎛️ ADD TRANDATA Options
Choose the right level for each scenario
🔑
Basic (Primary Key)
Logs PK columns only. Minimum required when a primary key exists. Best for standard OLTP tables.
ADD TRANDATA HR.EMPLOYEES
🗝️
NOKEY (Unique Index)
For tables without a primary key. Logs unique index columns or falls back to ROWID-based identification.
ADD TRANDATA HR.DEPT NOKEY
📋
ALLCOLS
Logs every column. Required for CDR (Conflict Detection & Resolution) in bi-directional replication.
ADD TRANDATA HR.EMPLOYEES ALLCOLS
🎯
KEYCOLS (Custom Key)
Define custom columns as the logical key. Overrides default key detection for legacy or composite scenarios.
ADD TRANDATA HR.EMP KEYCOLS (BADGE, SSN)
🏗️
SCHEMATRANDATA
Schema-level supplemental logging. Auto-applies to new tables added later. Includes scheduling columns for Parallel Replicat.
ADD SCHEMATRANDATA HR
⚡
PREPARECSN
Marks the SCN at which initial data load completed, enabling Extract to begin capture at the exact right point.
ADD TRANDATA HR.EMP PREPARECSN

💻 Local RAG Setup — Python (LangChain + FAISS + Llama / Oracle 26ai)
Python 3.10+
Step 1 — Install & Ingest
# Install dependencies pip install langchain faiss-cpu pip install sentence-transformers ollama pip install pypdf unstructured # Ingest GoldenGate docs from langchain.document_loaders import PyPDFLoader from langchain.text_splitter import RecursiveCharacterTextSplitter from langchain.embeddings import HuggingFaceEmbeddings from langchain.vectorstores import FAISS loader = PyPDFLoader("goldengate_21c_admin.pdf") docs = loader.load() splitter = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=64) chunks = splitter.split_documents(docs) embedder = HuggingFaceEmbeddings(model_name="sentence-transformers/all-MiniLM-L6-v2") vectorstore = FAISS.from_documents(chunks, embedder) vectorstore.save_local("gg_rag_index") print(f"Indexed {len(chunks)} chunks")
Step 2 — Query with Local LLM or Oracle 26ai
from langchain.llms import Ollama from langchain.chains import RetrievalQA vectorstore = FAISS.load_local("gg_rag_index", embedder, allow_dangerous_deserialization=True) # Option A: Local (Ollama / Llama3) llm = Ollama(model="llama3.2") # Option B: Oracle 26ai (via Anthropic SDK) # llm = Claude(model="claude-sonnet-4-6") qa_chain = RetrievalQA.from_chain_type( llm=llm, retriever=vectorstore.as_retriever(search_kwargs={"k": 5}), return_source_documents=True ) result = qa_chain.invoke( "What supplemental logging level is needed for Lakehouse?" ) print(result["result"])

🔴 Database-Level Prerequisites
ARCHIVELOG Mode
  • ✔Oracle DB must be in ARCHIVELOG mode
  • ✔GoldenGate mines archived redo logs for changes
  • ✔Check: SELECT LOG_MODE FROM V$DATABASE;
  • ✔Enable: SHUTDOWN → MOUNT → ALTER DATABASE ARCHIVELOG
Minimal Supplemental Logging
  • ✔Must be enabled at DB level before TRANDATA
  • ✔Requires SYSDBA or ALTER DATABASE privilege
  • ✔No instance restart required
  • ✔Works in RAC — apply once per DB, not per node
FORCE LOGGING
  • ✔Prevents NOLOGGING operations from bypassing redo
  • ✔Critical for direct path loads, CREATE TABLE NOLOGGING
  • ✔Apply at DB level AND PDB level for multitenant
  • ✔Check: SELECT FORCE_LOGGING FROM V$DATABASE;
Redo Log Sizing
  • ✔Minimum: 3 redo log groups, ≥ 500MB each
  • ✔Supplemental logging increases redo volume by 5–40%
  • ✔Target: log switches no more than 4 per hour
  • ✔Monitor: V$LOG_HISTORY for switch frequency

GoldenGate Sizing Guidelines
Based on Oracle internal sizing guidelines. Use these formulas to plan memory, storage, and network for your GoldenGate deployment.
🧮 Memory Requirement Calculator (Integrated Extract)
ℹ️
By default, one Extract requests the logmining server to run with MAX_SGA_SIZE of 1GB. For 3 Extracts in the same database instance, allocate at least 3 GB to the Streams pool. Best practice: keep 25% of Streams pool available.
Example: 3 Extracts → set STREAMS_POOL_SIZE = 3 GB × 1.25 = 3.75 GB
STREAMS_POOL_SIZE = (Number of Extracts × 1 GB) × 1.25
STREAMS_POOL_SIZE = 3.75 GB
SQL to Apply
ALTER SYSTEM SET STREAMS_POOL_SIZE = 3840M SCOPE=BOTH; -- Or in init.ora / spfile: STREAMS_POOL_SIZE=3840M
💾 Trail File Storage Calculator
ℹ️
Formula: ((transaction log size × 0.33) × log switches per day) × retention days
Example: 1GB logs, 10 switches/day, 7 days → 1 × 0.33 × 10 × 7 = 23 GB
Trail Storage = (Txn Log GB × 0.33) × Log Switches/Day × Retention Days
Trail Storage = 23.1 GB
