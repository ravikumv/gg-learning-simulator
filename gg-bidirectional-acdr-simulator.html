Architecture, prerequisites, Automatic Conflict Detection & Resolution (ACDR), Replicat selection, and a live conflict simulator

iWhat "3-site bi-directional" means

This is an active-active (multi-master) mesh: three databases — Site A, Site B, Site C — each accept live application writes on the same shared tables, and each site's changes are replicated to both of the other two sites. Unlike a hub-and-spoke or Option A/B mining topology, there is no single source of truth: every site is simultaneously a source and a target.

Because all three sites can be written to concurrently, this topology requires two things a simple one-way Extract→Replicat pair does not: loop prevention (so a change doesn't get re-captured and bounced back to where it came from) and conflict detection & resolution (so two near-simultaneous changes to the same row don't leave the sites permanently out of sync).

iiFull-mesh process topology

Oracle GoldenGate enforces a hard rule used throughout this simulator suite: one Replicat group delivers to exactly one target. For a 3-way mesh that means each site runs one local Extract and two outbound Data Pumps (one per peer), and receives two inbound Replicats (one per peer source trail) — six directional legs in total, not one shared Replicat fan-in.

Each Extract tags the transactions it captures (SETTAG). Every Extract is configured with TRANLOGOPTIONS EXCLUDETAG for the tag values written by local Replicats, so replicated data arriving from a peer and applied locally is never re-captured and sent back — this is what stops the 3-site mesh from ping-ponging changes in an endless loop.

iiiWhy 3 sites is harder than 2

  • 6 directional legs instead of 2 — process count and monitoring surface roughly triples versus a 2-site active-active pair.
  • Conflict windows multiply — a row can be touched at A, B, and C nearly simultaneously, not just at two points.
  • Sequence/identity collisions — auto-incrementing keys generated independently at 3 sites will collide unless each site is given a reserved, non-overlapping range or an identity strategy that encodes the site.
  • DDL propagation ordering — a DDL change issued at one site must reach the other two before conflicting DML against the new structure arrives, which is why DDL changes in an active-active mesh are typically applied off-peak and one site at a time.

1Database-level prerequisites (all 3 sites)

  • Database in ARCHIVELOG mode with FORCE LOGGING enabled.
  • Supplemental logging enabled at the database level, plus ADD TRANDATA (AdminClient) on every replicated table, capturing the primary key / unique index columns at minimum.
  • Every replicated table must carry a primary key or a unique index — this is a hard requirement for Auto CDR (ADD_AUTO_CDR will not operate on a table without one).
  • The LOGALLSUPCOLS Extract parameter set (default with integrated Extract) so all columns enabled for CDR comparison are actually written to the trail.
  • Time synchronization (NTP) across all 3 hosts — the default Auto CDR resolution method compares row timestamps, so clock drift directly causes wrong-winner resolutions.

2GoldenGate Microservices prerequisites

  • Deployment and Service Manager installed at all 3 sites; database registered with the deployment.
  • Integrated capture mode for Extract — required for Auto CDR, and mandatory anyway if any site is a multitenant/PDB database.
  • A conflict-safe key/sequence strategy agreed up front — e.g. reserved sequence cache ranges per site, or a site-tagged identity column — since GoldenGate itself does not arbitrate primary-key generation collisions.
  • SETTAG configured on each Replicat and matching TRANLOGOPTIONS EXCLUDETAG on the local Extract, to prevent apply-loop re-capture (see Architecture tab).

3Enabling ACDR itself

Auto CDR is enabled per table via a database package call — run on every participating site for that table:

EXEC DBMS_GOLDENGATE_ADM.ADD_AUTO_CDR(
       schema_name => 'APP',
       table_name  => 'CUSTOMER_ACCT');

This single call, run identically at Site A, Site B, and Site C, does three things at once: enables supplemental logging on all CDR-relevant columns, installs the conflict-detection comparison logic, and activates the default Latest-Timestamp resolution method — no manual COMPARECOLS/RESOLVECONFLICT parameters needed in the Replicat parameter file.

iWhat ACDR actually is

Automatic Conflict Detection and Resolution (ACDR) is an Oracle-Database-side feature (from Oracle Database 12.2 / GoldenGate 12.3.0.1 onward, refined through 21c) that manages conflict detection and resolution for you, via DBMS_GOLDENGATE_ADM.ADD_AUTO_CDR. It's distinct from — and replaces the need for — the older, manual approach: hand-writing COMPARECOLS and RESOLVECONFLICT clauses directly in the Replicat parameter file. That manual mechanism still exists and is still called Oracle GoldenGate CDR, but for a new active-active build ACDR is the recommended path.

Under the covers, GoldenGate represents each captured change as a row Logical Change Record (LCR) containing the operation type plus before/after column images. A hidden system column, CDRTS$ROW, stores the timestamp of the last change applied to that row so an incoming LCR can be compared against it at apply time.

iiConflict types ACDR detects

Conflict typeWhen it fires
INSERTROWEXISTSIncoming INSERT, but a row with that key already exists locally
UPDATEROWEXISTSIncoming UPDATE, but the local row's current values differ from what the source expected (classic lost-update)
UPDATEROWMISSINGIncoming UPDATE, but the target row no longer exists locally
DELETEROWEXISTSIncoming DELETE, but the local row has since changed
DELETEROWMISSINGIncoming DELETE, target row already gone

iiiDefault resolution: Latest Timestamp

With the default method, the row carrying the later CDRTS$ROW timestamp wins and is applied (or kept); the earlier one loses and is discarded. Set record_conflicts => TRUE on ADD_AUTO_CDR to have every WIN/LOSS decision logged for audit, which the Live Simulator tab visualizes.

This is why NTP synchronization across all 3 sites (called out in Prerequisites) isn't optional — a few seconds of clock drift can make an objectively older change look newer and silently overwrite a legitimate later edit.

iThe rule

Auto CDR only operates when Replicat is running in an "integrated flavor." That means either Integrated Replicat, or Parallel Replicat operating in integrated mode (sometimes written iPR). Classic/non-integrated Replicat and non-integrated Parallel Replicat cannot enforce Auto CDR at all — the feature depends on the inbound server the integrated modes use to construct and apply LCRs.

Oracle's general apply-mode guidance is to default to (non-integrated) Parallel Replicat for raw throughput unless a specific feature requires otherwise — and Auto CDR is exactly that exception for an active-active build.

⚙Inside the CDR engine — the two-phase check

Every inbound LCR goes through two distinct phases, and it's worth keeping them separate:

  • 1. Detection — the engine compares the operation the LCR is asking for (INSERT / UPDATE / DELETE) against the current state of the local row (exists or not). A mismatch between what the source expected and what's actually there is what produces a named conflict type.
  • 2. Resolution — only once a conflict type has been flagged does the configured method run. With Auto CDR's default, that's Latest Timestamp: compare the incoming LCR's CDRTS$ROW against the local row's CDRTS$ROW, and let the later one win.
Incoming opLocal row EXISTSLocal row MISSING
INSERTINSERTROWEXISTS → timestamp arbitratesNo conflict → plain insert
UPDATEUPDATEROWEXISTS → timestamp arbitratesUPDATEROWMISSING → discarded by default
DELETEDELETEROWEXISTS → timestamp arbitratesDELETEROWMISSING → no-op, already converged

Notice the asymmetry in the middle column: an *ROWEXISTS conflict is a fair fight decided by timestamp, but UPDATEROWMISSING is not timestamp-arbitrated — by default there is nothing to update, so the incoming change is simply discarded even if it's the "newer" one. This is a common gotcha in active-active designs: a DELETE can permanently defeat a slightly-later UPDATE from another site. (This simulator uses timestamp divergence as a teaching stand-in for the real detection check, which compares the LCR's captured before-image against the current column values.)

?What to notice

  • A single-site action propagates cleanly to both peers — no independent history to conflict with, so it's a routine apply.
  • In UPDATE vs UPDATE, both sites apply their own change locally first, then the later CDRTS$ROW wins everywhere once the peer's trail record arrives — the earlier one is logged as LOST, not silently dropped.
  • In INSERT vs INSERT, whichever insert a site sees first applies with no conflict (row was missing); the second insert to arrive at that same site is flagged INSERTROWEXISTS and arbitrated by timestamp.
  • In DELETE vs UPDATE, watch the asymmetry: the site that updated sees the incoming DELETE as a fair DELETEROWEXISTS fight (timestamp decides), but the site that deleted sees the incoming UPDATE as UPDATEROWMISSING and discards it outright — regardless of which timestamp is later.
  • In DELETE vs DELETE, there's no real contention at all — both sites already agree the row is gone, so incoming deletes just resolve as no-ops.
