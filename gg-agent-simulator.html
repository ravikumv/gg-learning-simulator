Design

Building an Agent for GoldenGate

An agent is not a bigger prompt. It's an LLM reasoning core wrapped in an orchestration loop, wired to real GoldenGate tools, with a memory of system state and a guardrail layer for anything destructive.

Reasoning Core (LLM)

Interprets the incoming task, forms a plan, and decides the next tool call based on what it has observed so far.

↓↑

Orchestrator — ReAct Loop

Runs Thought → Action → Observation → repeat, feeding each tool result back to the reasoning core until the task is resolved or escalated.

↓↑

Memory

Conversation history plus a cached snapshot of Extract/Replicat state, refreshed every cycle so decisions aren't made on stale status.

Guardrails

Any command that alters state — repositioning an Extract, restarting a Replicat, editing a parameter file — pauses for human approval first.

↓↑

Tool Layer → GoldenGate Microservices 26ai

The only way the agent touches the real environment. No tool, no action — this is what separates it from a chatbot.

Tools the agent actually needs

Each one wraps a real GoldenGate interface — nothing invented.

Handler .props ValidatorChecks target handler config (Snowflake, Iceberg, ADLS Gen2, Fabric, AIDP)

Alerting & NotificationRaises a ticket or page when a threshold breach is confirmed

The loop, in outline

// executed once per cycle until state == RESOLVED while (task.state != RESOLVED) { thought = llm.reason(task, memory.snapshot()) action = llm.chooseTool(thought) // guardrail check before anything that mutates state if (action.isDestructive) approval = awaitHumanApproval(action) observation = tools.call(action) memory.update(observation) } answer = llm.summarize(memory.history()) // grounded in real observations
⚠  Guardrails aren't optional polish — repositioning an Extract or restarting a Replicat can drop or duplicate data if the agent gets it wrong. Every state-changing action routes through approval before execution.
